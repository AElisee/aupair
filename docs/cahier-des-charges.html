1. Présentation du projet

AuPair A.EU est une plateforme web de mise en relation entre de jeunes Africains souhaitant devenir au pair et des familles d'accueil situées en Europe et en Amérique du Nord. Elle se positionne comme la première plateforme mondiale spécialisée sur ce segment.

Proposition de valeur

  • Pour les au pairs : accès à un annuaire de familles vérifiées, création d'un profil détaillé, messagerie directe, accompagnement vers une expérience à l'étranger.
  • Pour les familles : recherche d'au pairs qualifiés, profils vérifiés, inscription gratuite et illimitée.

Modèle économique

Abonnement 32 € / 30 jours côté au pair pour débloquer la messagerie. Inscription gratuite côté famille. Paiement prévu par carte bancaire / PayPal (Stripe) et Mobile Money (CinetPay) pour le marché africain.

Objectif de la mission de reprise

Le projet dispose aujourd'hui d'un front-end complet et abouti (toutes les pages, l'UI, la charte) mais le back-end est très largement à construire. La mission consiste à transformer ce prototype d'interface en une application réellement fonctionnelle : authentification, base de données réelle, API, messagerie, paiement, emails.

10. Feuille de route de finalisation

Approche recommandée : MVP d'abord (parcours utilisable rapidement), paiement réel et temps réel en dernier. Chaque phase est livrable et testable indépendamment.

Phase 0 — Base de données opérationnelle
  • Provisionner PostgreSQL : Postgres local pour le dev (Homebrew/Docker) + projet Supabase pour la prod.
  • Renseigner DATABASE_URL, puis npx prisma generate et npx prisma db push.
  • Créer prisma/seed.ts : 1 ADMIN, 2 au pairs + 2 familles (profils ACTIVE), articles de blog, FAQ, témoignages. Mots de passe hashés via bcryptjs.
Phase 1 — Authentification fonctionnelle (bloquant)
  • src/app/api/auth/[...nextauth]/route.ts exportant { GET, POST } = handlers (amorcé).
  • src/app/api/auth/register/route.ts : crée User + profil (PENDING), hash bcrypt, email unique (amorcé).
  • Brancher le stepper d'inscription (remplacer le TODO par un fetch POST puis signIn).
  • Ajouter <SessionProvider> dans le layout racine ; brancher l'état connecté + signOut dans la Navbar.
Phase 2 — Données réelles : annuaires & profils
  • Convertir les annuaires (/trouver-au-pair, /trouver-famille) en Server Components avec requêtes Prisma (findMany où status = ACTIVE), filtres via query params.
  • Pages de détail [id] (lecture + bouton « contacter », incrément profileViews).
  • Éditeur de profil au pair + famille branchés sur API /api/profils/... (GET/PUT).
  • Créer les 5 sous-pages manquantes du dashboard famille (profil, recherche, messages, notifications, paramètres).
  • KPIs des dashboards + Blog + FAQ alimentés par Prisma.
Phase 3 — Messagerie (polling puis temps réel)
  • API /api/messages (liste conversations, envoi) et /api/messages/[userId] (fil + marquage lu).
  • Gating abonnement réel côté serveur : helper hasActiveSubscription(userId) ; un au pair sans abonnement actif ne peut pas envoyer (les familles ne sont pas restreintes).
  • UI branchée avec polling (~5 s), puis migration vers Supabase Realtime.
Phase 4 — Abonnement
  • API /api/abonnement : activation (création Subscription ACTIVE, +30 jours) et lecture (jours restants).
  • Étape transitoire possible : activation simulée pour tester le gating avant l'intégration paiement réelle.
Phase 5 — Contact & back-office admin
  • Formulaire contact → création d'un Ticket.
  • Brancher les pages admin (utilisateurs, modération, paiements) sur Prisma ; API d'actions (valider / masquer / supprimer) protégées par vérification role === "ADMIN".
Phase 6 — Upload photos (Supabase Storage)
  • Bucket Supabase Storage, API /api/upload (multipart), stockage des URLs dans profilePhotoUrl / familyPhotoUrl / idDocumentUrl.
Phase 7 — Paiement réel & emails (post-MVP)
  • Stripe Checkout + webhook (/api/webhooks/stripe) ; CinetPay pour le Mobile Money.
  • Resend : emails transactionnels (bienvenue, confirmation d'abonnement, rappels d'expiration).

12. Points d'attention techniques

Prisma 7
  • Le datasource du schema.prisma n'a pas de champ url (nouveauté v7) ; l'URL est dans prisma.config.ts.
  • Importer le client depuis @/generated/prisma/client (et non @prisma/client).
  • Utiliser l'adapter PrismaPg (déjà fait dans src/lib/prisma.ts).
  • Après tout changement de schéma : npx prisma generate puis npx prisma db push.
NextAuth v5
  • Importer auth, signIn, signOut, handlers depuis @/lib/auth.
  • Dans une route API ou un Server Component : const session = await auth().
  • La stratégie de session est JWT ; le rôle et l'id sont propagés via les callbacks.
Next.js App Router — règle Client / Server
  • Une page qui passe des composants React (ex. icônes Lucide) en props à un Client Component doit être "use client".
  • Une page "use client" ne peut pas exporter metadata → créer un wrapper Server Component si besoin de SEO.
Bonnes pratiques attendues : valider toutes les entrées API avant écriture (zod recommandé), réutiliser l'instance Prisma partagée de src/lib/prisma.ts, vérifier role === "ADMIN" côté serveur sur les routes admin, ne jamais faire confiance au gating uniquement côté UI.
